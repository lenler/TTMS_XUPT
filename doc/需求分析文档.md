# TTMS 需求分析文档

## 1. 文档目的

本文档用于说明 TTMS 剧院票务管理系统的业务目标、用户角色、功能需求和验收标准，为前端、后端、数据库设计及测试工作提供统一依据。

## 2. 系统目标

TTMS 面向剧院日常运营，支持演出厅、剧目、演出计划、售票、退票、验票、员工、观众和财务统计等业务。系统采用前后端分离模式，前端先通过 Mock 接口完成页面联调，后续与服务端接口对接。

## 3. 用户角色

| 角色 | 主要职责 |
| --- | --- |
| 系统管理员 | 维护演出厅、剧目、员工、角色权限等基础数据 |
| 票务人员 | 查询演出计划，完成线下售票、退票和验票操作 |
| 财务人员 | 查看票房、退款、销售排名和员工业绩 |
| 观众用户 | 浏览影片、选座购票、查看订单和发起退票 |

## 4. 演出厅管理需求

### 4.1 业务目标

演出厅是剧目排期、座位生成和售票的基础数据。管理员需要维护每个演出厅的名称、座位行列规模、简介和启用状态，保证后续演出计划可以准确关联演出厅并生成票务座位。

### 4.2 功能需求

| 编号 | 功能 | 说明 |
| --- | --- | --- |
| STUDIO-01 | 演出厅列表查询 | 支持分页查看演出厅列表，展示名称、座位规模、座位总数、状态和简介 |
| STUDIO-02 | 关键字搜索 | 支持按演出厅名称或简介进行模糊搜索 |
| STUDIO-03 | 新增演出厅 | 管理员可填写名称、座位行数、座位列数和简介创建演出厅 |
| STUDIO-04 | 修改演出厅 | 管理员可修改已有演出厅的基础信息 |
| STUDIO-05 | 删除演出厅 | 管理员可删除暂未使用或录入错误的演出厅 |
| STUDIO-06 | 查看详情 | 管理员可查看单个演出厅的完整信息，为后续座位维护扩展预留入口 |

### 4.3 数据项

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| id | number | 是 | 演出厅唯一编号，由系统生成 |
| name | string | 是 | 演出厅名称，建议不超过 30 个字符 |
| rowCount | number | 是 | 座位行数，当前前端限制 1 到 30 |
| colCount | number | 是 | 座位列数，当前前端限制 1 到 40 |
| introduction | string | 是 | 演出厅简介，说明用途、设备或特点 |
| status | number | 是 | 状态，1 表示启用，0 表示停用 |

### 4.4 业务规则

1. 演出厅名称、座位行数、座位列数和简介不能为空。
2. 座位总数由 `rowCount * colCount` 自动计算，不由用户手动录入。
3. 删除演出厅前应由用户二次确认，避免误删基础数据。
4. 后续接入真实后端时，若演出厅已经关联演出计划，后端应禁止物理删除，可改为停用。
5. 当前阶段使用 MSW Mock 完成本地开发，接口路径遵循《接口设计文档.md》中的 `/admin/api/studios` 约定。

### 4.5 异常处理

| 场景 | 处理方式 |
| --- | --- |
| 表单必填项为空 | 前端表单提示具体字段错误 |
| 查询不到详情 | 展示空状态，不阻塞用户返回列表 |
| 删除不存在的演出厅 | Mock 接口返回失败信息，前端统一错误提示 |
| 网络请求失败 | Axios 拦截器统一提示“网络请求失败” |

### 4.6 验收标准

1. 管理员登录后进入 `/admin/studio`，可以看到演出厅分页表格。
2. 输入关键字并搜索，可以过滤演出厅名称或简介。
3. 点击“新增演出厅”，填写合法数据并保存后，列表出现新记录。
4. 点击“编辑”，修改信息并保存后，列表数据同步更新。
5. 点击“删除”并确认后，目标记录从列表中移除。
6. 点击“详情”，可以进入 `/admin/studio/:id` 查看演出厅详情。

## 5. 后续需求补充
