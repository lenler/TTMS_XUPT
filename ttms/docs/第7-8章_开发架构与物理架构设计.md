# 7. 开发架构设计

本节从工程组织、源代码构成以及系统组件关系三个方面，对 TTMS（剧院票务管理系统）的开发架构进行说明。结合实际代码实现情况，系统采用前后端分离的开发模式：前端使用 React、TypeScript 与 Vite 构建单页应用，后端使用 Spring Boot 与 MyBatis 实现业务服务，数据库采用 MySQL 存储核心业务数据。整个系统在开发层面形成了较为清晰的分层结构，能够支撑管理端和观众端两类业务场景的协同实现。

从软件工程角度看，TTMS 的开发架构具有以下特点：一是前后端职责边界明确，前端主要承担界面呈现、用户交互和请求组织，后端主要承担业务处理、数据访问和统一响应；二是后端采用典型的分层架构，控制层、服务层、数据访问层和领域模型层职责清晰；三是前端按页面、路由、状态、服务和组件进行组织，便于功能扩展与协同开发；四是数据库脚本与设计文档独立管理，有利于环境初始化、版本演进和课程设计材料整理。

## 7.2 源代码文件

本小节说明项目工程中的主要源代码文件构成，以及这些文件与软件逻辑单元之间的对应关系。由于系统采用前后端分离开发模式，因此源代码构成也分为前端源代码、后端源代码以及辅助性的数据库脚本和文档文件三部分。

### 7.2.1 前端源代码构成

前端采用 React 单页应用模式实现，系统启动入口位于 `src/main.tsx`，由根组件 `App.tsx` 统一挂载管理端与观众端路由。整个前端工程不仅承担界面展示功能，还负责路由管理、接口调用、权限守卫和前端状态管理等工作。前端主要源代码与逻辑单元关系如表 7-2 所示。

表 7-2 前端主要源代码文件与逻辑单元关系

| 源码目录/文件 | 逻辑单元 | 说明 |
| --- | --- | --- |

| `src/App.tsx` | 根路由组织单元 | 聚合管理端与观众端路由，完成全局主题配置与路由挂载。 |

| `src/routes/customer.routes.tsx` | 观众端路由单元 | 定义观众端页面路由，包括首页、排期查询、选座购票、订单、钱包与个人中心等模块。 |

结合前端页面组织方式，可以将其划分为两个相对独立但共享基础设施的子系统：

1. 管理端子系统。主要面向管理员、售票员、检票员等后台角色，完成剧目管理、演出厅管理、演出计划管理、现场售票、退票处理、验票管理、顾客和员工管理以及财务统计等业务。
2. 观众端子系统。主要面向普通顾客，完成剧目浏览、场次查询、在线选座、下单支付、订单查询、钱包充值和个人信息维护等业务。

从实现方式上看，前端通过统一请求模块与两套路由配置实现“同一代码仓库、双端业务共存”的开发模式。这种方式既减少了重复开发成本，又使管理端和观众端能够共享公共组件、类型定义与请求基础设施。

### 7.2.2 后端源代码构成

后端采用 Spring Boot 分层架构实现，以 Java 17 作为主要开发语言，利用 MyBatis 完成数据库访问。后端的启动入口为 `TtmsApplication.java`，通过 `@MapperScan` 自动扫描数据访问层接口，并由 Spring 容器统一管理控制器、服务和配置类。后端主要源代码文件与逻辑单元关系如表 7-3 所示。

表 7-3 后端主要源代码文件与逻辑单元关系

| 源码目录/文件 | 逻辑单元 | 说明 |
| --- | --- | --- |
| `TtmsApplication.java` | 系统启动单元 | 启动 Spring Boot 应用，并完成数据访问层扫描与组件装配。 |
| `controller/` | 控制层单元 | 接收前端请求，完成参数绑定、业务调用和结果返回。 |
| `service/` | 业务接口单元 | 定义剧目管理、排期管理、售票、验票、财务等核心业务接口。 |
| `service/impl/` | 业务实现单元 | 实现订单支付、票状态流转、排期冲突检测和自动生成票等核心逻辑。 |
| `repository/` | 数据访问单元 | 基于 MyBatis 注解 SQL 与数据库交互，实现数据的增删改查。 |

从业务角度看，后端代码又可以抽象为若干核心逻辑单元：

1. 用户认证单元。负责管理员登录、观众登录与注册、身份识别以及基础权限处理。
2. 票务处理单元。负责锁票、下单、支付、退票、验票等完整票务流程，是系统最核心的业务部分。
3. 场次管理单元。负责演出计划创建、时间冲突检测，以及排期创建后的票据自动生成。
4. 基础资料管理单元。负责剧目、演出厅、座位、员工、顾客、角色等基础数据维护。
5. 财务统计单元。负责销售记录汇总、收入统计、员工销售额分析和剧院经营数据统计。

后端在接口暴露上还采用了“业务接口 + 前端兼容接口”的组织方式。其中，管理端兼容接口集中在 `/admin/api/*` 路径下，观众端兼容接口集中在 `/customer/api/*` 路径下。这样可以使前端以统一的数据格式访问后端，同时复用同一套业务服务层实现。

按照职责划分，系统组件可以归纳为以下几类。

### 7.3.1 表现层组件

表现层组件主要由管理端前端和观众端前端构成，是系统面向最终用户的直接交互层。

1. 管理端前端组件。主要为管理员、售票员、检票员等后台角色提供操作界面，用于执行剧目管理、排期管理、票务处理、用户维护和财务统计等操作。
2. 观众端前端组件。主要为普通顾客提供在线浏览剧目、查询场次、选择座位、提交订单、完成支付和管理个人订单等功能。
3. 路由与布局组件。负责页面导航、权限守卫、菜单呈现与页面框架控制，是前端界面组织的重要基础设施。

### 7.3.2 控制与业务组件

控制与业务组件位于系统后端，是整个系统处理流程的核心。

1. 控制层组件。负责接收前端请求、解析参数、调用服务层并统一封装响应结果。
2. 兼容接口组件。对外提供 `/admin/api/*` 和 `/customer/api/*` 两类接口路径，以满足管理端和观众端的不同调用习惯。
3. 业务服务组件。包括认证服务、排期服务、票务服务、验票服务和财务服务等，是系统实现业务规则和业务流程的核心。
4. 公共处理组件。负责统一异常处理、统一响应结构以及跨域规则配置，保证接口行为的一致性。

### 7.3.3 数据访问组件

数据访问组件负责业务对象与数据库之间的映射与交互，主要由 Repository 和领域模型组成。

1. Repository 组件。通过 MyBatis 注解 SQL 完成对数据库的访问，承担数据的增删改查操作。
2. Domain 组件。对剧目、演出厅、座位、场次、票、销售单、顾客、员工和角色等核心业务对象进行抽象，是业务逻辑与数据结构之间的桥梁。
3. DTO 组件。承担控制层与服务层、前端与后端之间的数据传递职责，保证接口输入输出清晰、结构统一。

### 7.3.4 数据存储组件

数据存储组件主要为 MySQL 数据库，负责存储系统运行过程中产生的各类业务数据，包括剧目数据、场次数据、座位数据、票数据、订单数据、顾客数据、员工数据及权限数据等。数据库是整个系统的持久化基础，也是统计分析和业务追踪的重要数据来源。

### 7.3.5 组件协作关系说明

系统各组件之间的协作过程可以概括为以下步骤：

1. 用户通过浏览器访问管理端或观众端页面。
2. 前端页面根据用户操作向后端对应接口发送 HTTP 请求。
3. 控制层组件接收请求后完成参数绑定，并调用相应业务服务。
4. 业务服务组件根据业务规则执行业务处理，例如锁票、支付、退票或统计计算。
5. 业务服务通过数据访问组件完成数据库读写操作。
6. 数据库返回处理结果后，系统将其逐层封装并返回给前端页面进行展示。

从上述过程可以看出，TTMS 的开发架构整体呈现出较为标准的多层结构。各层之间依赖关系清晰、职责边界明确，便于后续在不影响整体结构的前提下对某一层进行局部优化或功能扩展。

# 8. 物理架构设计

本节从网络环境和部署方案两个方面说明系统的物理架构。结合现有代码配置、项目运行方式以及公网访问验证结果，TTMS 当前部署在华为云服务器环境中，对外由 Nginx 提供统一访问入口，后端业务服务运行在应用端口，数据库运行在服务器本机或同主机内部网络环境中。该部署方式能够满足课程设计项目在展示、联调与验收阶段的使用需求。

## 8.2 部署方案

### 8.2.1 部署总体思路

TTMS 当前采用单机部署方案，即前端静态资源、后端业务服务和数据库服务统一部署在同一台华为云服务器中，由 Nginx 作为系统统一访问入口。该部署方式结构简单、成本较低、搭建效率高，适合课程设计系统的开发、联调、演示和验收。

在该部署方案下，系统运行的基本思路如下：

1. 前端工程通过 Vite 进行打包构建，生成可直接部署的静态资源文件。
2. 构建完成后的前端文件部署至 Nginx 所管理的站点目录，由 Nginx 直接向浏览器返回页面资源。
3. 后端工程通过 Maven 打包生成可执行 Jar 包，并以 Java 进程方式运行。
4. Nginx 根据访问路径判断请求类型：若为页面请求，则返回静态资源；若为接口请求，则将其反向代理至 Spring Boot 应用。
5. Spring Boot 应用根据业务逻辑访问 MySQL 数据库，完成数据查询、更新和事务处理。

这种部署模式避免了前端开发服务器在生产环境中继续运行，也使后端服务端口无需直接暴露给公网，从而提升了系统结构的清晰性和部署的规范性。

### 8.2.3 部署过程说明

系统部署过程可以分为前端部署、后端部署和数据库部署三个部分。

1. 前端部署。前端工程执行构建命令后生成静态文件，部署到 Nginx 所管理的目录中。用户访问系统首页时，Nginx 直接返回前端页面和相关静态资源。
2. 后端部署。后端工程通过 Maven 打包为可执行 Jar 包，在服务器上以 Java 进程方式运行，监听 `8080` 端口。后端应用负责处理管理端和观众端的业务接口请求。
3. 数据库部署。MySQL 数据库与后端服务部署在同一台服务器或同一主机内部环境中，后端通过 JDBC 使用本地地址进行连接，以减少网络访问开销。
4. 代理部署。Nginx 负责将外部请求进行分类转发，其中页面资源由 Nginx 直接处理，接口请求则由 Nginx 转发给后端应用处理。

整个部署流程中，Nginx 起到了连接浏览器请求与应用服务的桥梁作用，Spring Boot 起到了业务逻辑中心的作用，MySQL 则负责持久化存储。三者共同构成完整的运行环境。

### 8.2.4 Nginx 代理策略

结合代码结构与部署方式，Nginx 在系统中的主要职责包括以下几个方面。

1. 代理根路径 `/`，返回前端构建后的静态页面。
2. 转发观众端接口路径 `/customer/api/*` 到后端 Spring Boot 服务。
3. 转发管理端接口路径 `/admin/api/*` 到后端 Spring Boot 服务。
4. 处理前端单页应用在浏览器刷新场景下的路由回退问题。
5. 为后续扩展 HTTPS、缓存控制和访问日志管理提供统一入口。

因此，在实际运行中可以将 Nginx 的工作逻辑理解为：

- 页面类请求由 Nginx 直接从磁盘读取静态文件并返回；
- 接口类请求由 Nginx 反向代理至 `http://127.0.0.1:8080`；
- 后端业务服务再通过 JDBC 与 `localhost:3306` 上的 MySQL 进行交互。

### 8.2.5 部署方案特点分析

本系统当前部署方案具有以下优点。

1. 架构简单清晰，便于搭建、排查问题和进行课程设计演示。
2. 前后端职责明确，页面资源和业务接口的访问路径统一由 Nginx 管理。
3. 后端端口无需直接暴露给外部用户，提高了部署组织性和可维护性。
4. 前端静态化部署后，访问效率较高，资源管理相对简单。
5. 单机部署成本较低，适合项目当前的访问规模和实验环境。

当然，该部署方案也存在一定局限性，例如前端、后端和数据库部署在同一台服务器上，系统资源彼此竞争；当访问量增大时，单机承载能力有限；锁票会话等临时状态目前仍偏向单实例实现，不利于横向扩展。因此，该方案更适合作为课程设计、原型系统和中小规模业务的部署方案。

### 8.2.6 后续扩展建议

如果系统后续需要面向更高并发访问或作为正式业务系统运行，可以在现有物理架构基础上逐步演进，主要方向如下。

1. 将前端静态资源迁移到对象存储或 CDN，提升资源分发效率。
2. 将 MySQL 从应用服务器中独立出来，迁移为云数据库实例，降低单机耦合度。
3. 将锁票会话、缓存数据等迁移到 Redis，以提升并发处理能力和状态共享能力。
4. 将 Nginx 与应用服务拆分部署，并通过负载均衡实现多实例访问。
5. 为系统增加 HTTPS、日志审计、监控告警与自动备份机制，提高系统的安全性与可运维性。

综上所述，TTMS 当前采用的物理架构能够满足项目现阶段的运行需求，且具备较好的可理解性和实现可行性。对于课程设计报告而言，该部署方案既能够体现完整的软件工程实现过程，也能够为后续进一步优化和扩展提供清晰基础。
