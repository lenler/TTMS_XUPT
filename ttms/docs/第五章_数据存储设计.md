# 第五章 数据存储设计

## 5.1 内部数据结构

系统在运行过程中使用内存数据结构存储临时状态信息，主要包括锁座会话缓存和验票记录缓存。

### 5.1.1 锁座会话（LockSession）

锁座会话用于管理观众选座后的临时锁座状态，采用 `ConcurrentHashMap` 存储。键是 `lockToken` 字符串，值是 `LockSession` 对象。代码位于 `CustomerCompatController.java` 中：

```java
private final ConcurrentHashMap<String, LockSession> lockSessions = new ConcurrentHashMap<>();
```

LockSession 内部类包含两个字段：`token`（锁座令牌，UUID 字符串，全局唯一标识一次锁座操作）和 `tickets`（本次锁座涉及的票据对象列表）。

锁座会话的生命周期分为四个阶段：

- **创建**：观众调用锁座 API（POST /customer/api/orders/lock）时，系统为该请求生成唯一 lockToken，创建 LockSession 实例，连同锁定的票据一起存入 lockSessions 缓存。
- **读取**：调用下单 API（POST /customer/api/orders）时，通过 lockToken 从缓存中取出会话，获取已锁定的票据信息。
- **释放**：下单完成后立即释放；锁定时长超过 10 分钟（由配置文件 ttms.ticket.lock-timeout-minutes 控制）自动释放。
- **删除**：下单成功后调用 `lockSessions.remove(lockToken)` 清除缓存记录。

### 5.1.2 验票记录缓存（CheckRecord）

验票记录用于暂存管理端验票操作的历史记录，采用 `ArrayList` 内存列表存储，代码位于 `AdminTicketCompatController.java` 中：

```java
private final List<AdminCheckRecord> checkRecords = new ArrayList<>();
```

AdminCheckRecord 包含以下字段：id（验票记录 ID）、ticketId（已验票据 ID）、seatRow（座位行号）、seatCol（座位列号）、playName（剧目名称）、studioName（演出厅名称）、showTime（演出时间）、verifyTime（验票时间）、operatorName（操作员名称）、result（验票结果，取值为"验票通过"或"验票失败"）。

> 说明：当前版本使用内存列表存储验票记录，数据在服务重启后丢失。后续版本将接入关系型数据库持久化。

### 5.1.3 全局变量汇总

代码中定义了两个全局内存变量：`lockSessions`（类型 `ConcurrentHashMap<String, LockSession>`，位于 `CustomerCompatController`，用途是锁座会话缓存）和 `checkRecords`（类型 `List<AdminCheckRecord>`，位于 `AdminTicketCompatController`，用途是验票记录缓存）。

## 5.2 数据库

系统采用 MySQL 5.7 关系型数据库进行数据持久化存储，数据库名称为 `ttms`，字符集使用 `utf8mb4`。

### 5.2.1 数据库设计

系统共包含 14 张数据库表。表之间的关系如下：

核心业务表共 6 张：studios（演出厅）、plays（剧目）、schedules（演出计划/排片）、tickets（票据/座位票）、sales（销售单/订单）、sale_items（销售明细/票项）。

辅助表共 4 张：seats（座位）、customers（观众用户）、customer_recharges（充值记录）、employees（员工）。

权限表共 4 张：roles（角色）、resources（权限资源）、role_resources（角色-资源关联）、employee_roles（员工-角色关联）。

表之间的关联关系如下：

studios 和 schedules 是一对多关系，通过 studio_id 外键关联：一个演出厅可以有多个排片，一个排片属于一个演出厅。

plays 和 schedules 是一对多关系，通过 play_id 外键关联：一个剧目可以有多个排片，一个排片属于一个剧目。

schedules 和 tickets 是一对多关系，通过 schedule_id 外键关联：一个排片可以生成 64 张座位票（8 行 x 8 列），每张票对应一个排片。

studios 和 seats 是一对多关系，通过 studio_id 外键关联：一个演出厅包含 64 个座位，每个座位属于一个演出厅。

tickets 和 sale_items 是一对多关系，通过 ticket_id 外键关联：一张票对应一条销售明细，一条明细对应一张票。

sales 和 sale_items 是一对多关系，通过 sale_id 外键关联：一个订单包含多条明细（每人可买多张票），每条明细属于一个订单。

customers 和 sales 是一对多关系，通过 customer_id 外键关联：一个观众可以有多笔订单，一笔订单属于一个观众。

customers 和 customer_recharges 是一对多关系，通过 customer_id 外键关联：一个观众有多条充值记录。

employees 和 sales 是一对多关系，通过 employee_id 外键关联：一个员工可处理多笔柜台销售。

employees 和 roles 是多对多关系，通过中间表 employee_roles 关联，一个员工可以有多个角色，一个角色可以赋予多个员工。

roles 和 resources 是多对多关系，通过中间表 role_resources 关联，一个角色可以访问多个资源，一个资源可被多个角色访问。

seats 和 tickets 是一对多关系，通过 seat_id 外键关联：一个座位在不同场次对应不同的票，每张票指向一个座位。

### 5.2.2 数据表定义

**studios（演出厅）**

包含 id（主键，自增）、name（演出厅名称，非空）、row_count（座位行数，非空）、col_count（座位列数，非空）、introduction（演出厅简介，可空）、status（状态，默认 ACTIVE，可选 DISABLED）、created_at（创建时间）、updated_at（更新时间）。

**seats（座位）**

包含 id（主键，自增）、studio_id（所属演出厅，外键指向 studios）、row_no（行号，非空）、col_no（列号，非空）、status（座位状态，默认 ACTIVE）、created_at、updated_at。存在唯一约束（studio_id, row_no, col_no），保证同一演出厅内行号列号不重复。

**plays（剧目）**

包含 id（主键，自增）、type（剧目类型，如话剧/京剧/音乐剧等）、language（演出语言）、name（剧目名称）、introduction（简介，可空）、poster_url（海报路径，可空）、trailer_url（预告片链接，可空）、duration_minutes（演出时长，分钟）、base_price（基础票价）、status（状态，默认 ACTIVE）、created_at、updated_at。

**schedules（演出计划/排片）**

包含 id（主键，自增）、studio_id（演出厅 ID，外键）、play_id（剧目 ID，外键）、show_time（演出时间）、ticket_price（本场票价，可覆盖剧目基础票价）、status（状态，默认 ACTIVE）、created_at、updated_at。建有索引 idx_schedule_play（play_id）和 idx_schedule_studio_time（studio_id, show_time）以加速按剧目和按时间的查询。

**tickets（票据/座位票）**

包含 id（主键，自增）、seat_id（座位 ID，外键）、schedule_id（排片 ID，外键）、price（该票销售价）、status（票据状态，默认 AVAILABLE）、lock_time（锁座时间，可空）、version（乐观锁版本号，默认 0）、created_at、updated_at。

存在唯一约束（schedule_id, seat_id），表示同一场次同一座位只能有一张票。建有索引 idx_ticket_schedule_status（schedule_id, status）加速按排片和状态的查询。

票据状态有六种取值：AVAILABLE（可用，初始状态，可被锁定和购买）、LOCKED（已锁定，用户选座后暂锁，超时自动释放）、SOLD（已售，支付完成，不可再选）、CHECKED（已验票，入场验票通过）、REFUNDED（已退票，退票后恢复为可选）、VOIDED（已作废，管理端操作）。

**customers（观众用户）**

包含 id（主键，自增）、username（用户名，唯一）、password_hash（密码）、name（真实姓名，可空）、phone（手机号，可空）、email（邮箱，可空）、gender（性别，0=未知 1=男 2=女）、payment_password（支付密码，可空）、balance（账户余额，默认 0）、status（状态，默认 ACTIVE）、created_at（注册时间）、updated_at（更新时间）。

**customer_recharges（充值记录）**

包含 id（主键，自增）、customer_id（观众 ID，外键）、amount（充值金额）、balance_after（充值后余额）、recharge_time（充值时间）、created_at、updated_at。建有索引 idx_recharge_customer_time（customer_id, recharge_time）加速按客户和时间查询。

**employees（员工）**

包含 id（主键，自增）、employee_no（工号，唯一）、name（姓名）、position（岗位，如系统管理员/售票员）、phone（电话，可空）、email（邮箱，可空）、password_hash（登录密码）、status（状态，默认 ACTIVE）、created_at、updated_at。

**sales（销售单/订单）**

包含 id（主键，自增）、employee_id（操作员工，可空，外键，柜台售票时填写）、customer_id（观众 ID，可空，外键，在线购票时填写）、sale_time（下单时间）、paid_amount（实付金额）、change_amount（找零金额，柜台现金支付时使用）、sale_type（销售类型，取值为 ONLINE 或 COUNTER）、status（订单状态）、created_at、updated_at。建有索引 idx_sale_time（sale_time）加速按时间查询。

订单状态有四种取值：PENDING_PAYMENT（待支付，订单已创建尚未支付）、PAID（已支付，款项结清票已售出）、REFUNDED（已退票）、CANCELLED（已取消）。

销售类型有两种：ONLINE（观众端在线购买）和 COUNTER（柜台现场售票）。

**sale_items（销售明细/票项）**

包含 id（主键，自增）、sale_id（所属订单，外键）、ticket_id（所属票据，外键）、price（该票实际售价）、created_at、updated_at。

**roles（角色）**

包含 id（主键，自增）、name（角色名称，唯一）、created_at、updated_at。

**resources（权限资源）**

包含 id（主键，自增）、type（资源类型，menu 或 button）、name（资源名称）、url（资源路径）、created_at、updated_at。

**role_resources（角色-资源关联）**

包含 role_id（角色 ID，联合主键，外键）和 resource_id（资源 ID，联合主键，外键）。

**employee_roles（员工-角色关联）**

包含 employee_id（员工 ID，联合主键，外键）和 role_id（角色 ID，联合主键，外键）。

### 5.2.3 核心业务数据流

以下以在线订票流程为例，说明数据在数据库表中的流转：

首先，后台创建排片时向 schedules 表插入一条记录，关联 studios 和 plays。随后系统自动为这场排片生成 64 张座位票（8 行 x 8 列），向 tickets 表插入 64 条记录，状态均为 AVAILABLE。

观众在网站上选座后，系统将选中的票状态由 AVAILABLE 改为 LOCKED，同时记录 lock_time 为当前时间，并将锁座信息存入内存的 lockSessions 缓存中。

观众确认下单时，系统先向 sales 表插入一条记录，状态为 PENDING_PAYMENT，再向 sale_items 表插入对应条数的明细记录，关联 sale 和 ticket。

观众确认支付后，系统执行三个更新：将 tickets 表中对应票的状态由 LOCKED 改为 SOLD（同时乐观锁版本号 version 加 1），将 sales 表的状态改为 PAID 并填入实际支付金额，将 customers 表的 balance 减去票价金额。

如果观众申请退票，系统执行三个更新：将 tickets 表的状态改为 REFUNDED，将 sales 表的状态改为 REFUNDED，将 customers 表的 balance 加回退款金额。

## 5.3 数据文件

系统涉及以下几类数据文件。

### 5.3.1 配置文件

**application.yml（后端 Spring Boot 配置）**

位于 `ttms/backend/src/main/resources/application.yml`，用于配置 Spring Boot 服务器参数、数据库连接、CORS 跨域设置和业务参数。主要配置项包括：server.port（后端监听端口，默认 8080）、server.servlet.context-path（接口路径前缀，默认 /api）、spring.datasource.url（MySQL 连接地址，格式 jdbc:mysql://localhost:3306/ttms）、spring.datasource.username（数据库用户名，默认 ttms）、spring.datasource.password（数据库密码，默认 ttms）、ttms.cors.allowed-origins（CORS 跨域白名单列表）以及 ttms.ticket.lock-timeout-minutes（锁座超时时间，默认 10 分钟）。

**前端构建配置**

位于 `ttms/frontend/.env` 和 `ttms/frontend/vite.config.ts`，用于配置前端开发服务器和 API 代理规则。主要配置项包括：VITE_ENABLE_MOCK（是否启用 Mock 模式，默认 true）、server.port（前端开发端口，默认 3000）和 server.proxy（API 代理规则，将 /admin/api 路径代理到 localhost:8080/api）。

### 5.3.2 日志文件

**后端运行日志**位于 `ttms/backend/backend-h2-run.log`（H2 内存数据库模式下使用），记录 Spring Boot 应用程序运行过程中的信息、警告和错误。日志格式为标准 Spring Boot 格式，例如：

```
2026-06-08T11:30:13.378+08:00  INFO --- [ttms-backend] c.h.t.TtmsApplication : Starting TtmsApplication v0.1.0
```

**PM2 进程管理日志**记录后端 Java 进程在 PM2 管理下的输出。标准输出日志位于 `~/.pm2/logs/ttms-backend-out.log`，错误输出日志位于 `~/.pm2/logs/ttms-backend-error.log`。

### 5.3.3 图片资源文件

剧目海报等静态图片资源存储在服务器 `/var/www/html/theater_ticket/images/` 目录下，通过 Nginx 直接对外提供访问。目录按上传日期分层组织，例如 `/var/www/html/theater_ticket/images/2026-06-10/menghuitang.jpg`，访问方式为 `http://域名/images/2026-06-10/menghuitang.jpg`。

### 5.3.4 部署脚本

服务器端部署脚本位于 `deploy.sh`，执行一键部署操作。脚本依次执行：git pull origin main（拉取最新代码）、mvn clean package -DskipTests（编译后端）、pnpm run build（构建前端）、cp -rf dist/* /var/www/html/theater_ticket/（部署前端静态文件）、pm2 restart ttms-backend（重启后端服务）。
