# TTMS 接口设计文档

## 1. 文档说明

本文档定义 TTMS 前后端交互的全部 RESTful API 接口。前端采用 Mock 驱动开发模式，后端接口未就绪前使用 Mock 数据独立开发。

> **与后端内部接口的关系**：后端在 `/api/*` 下另有一套业务 REST 接口（如 `/api/studios`、`/api/sales/orders` 等），返回格式为 `{ code, message, data }`。本文档定义的前端兼容接口（`/admin/api/*`、`/customer/api/*`）通过兼容 Controller 层调用同一 Service 层，对外返回 `{ resCode, resMsg, data }` 格式。前端开发只需关注本文档中的兼容接口。

## 2. 通用约定

### 2.1 基础路径

| 端 | Base URL |
| --- | --- |
| 管理端 API | `/admin/api` |
| 观众端 API | `/customer/api` |

### 2.2 统一响应格式

```json
{
  "resCode": "10000",
  "resMsg": "请求成功",
  "data": {}
}
```

### 2.3 响应码

| resCode | 含义 |
| --- | --- |
| `10000` | 成功 |
| `20001` | 参数错误 |
| `20002` | 未登录 |
| `20003` | 无权限 |
| `20004` | 数据不存在 |
| `20005` | 业务规则限制（如座位已售、时间冲突等） |
| `30001` | 服务器内部错误 |

### 2.4 分页请求参数

```json
{
  "page": 1,
  "pageSize": 10,
  "keyword": ""
}
```

### 2.5 分页响应格式

```json
{
  "resCode": "10000",
  "resMsg": "请求成功",
  "data": {
    "list": [],
    "total": 100,
    "page": 1,
    "pageSize": 10
  }
}
```

### 2.6 核心状态枚举

以下状态枚举是整个票务链路的基础，所有接口必须遵守。

#### 2.6.1 票状态（Ticket）

| 值 | 标识 | 含义 | 可转换至 |
| --- | --- | --- | --- |
| `0` | `available` | 可售 | `locked` |
| `1` | `locked` | 已锁定（待支付） | `available`（超时释放）、`sold` |
| `2` | `sold` | 已售 | `checked`、`refunding` |
| `3` | `checked` | 已验票/已入场 | —（终态） |
| `4` | `refunding` | 退票处理中 | `refunded` |
| `5` | `refunded` | 已退票 | `available`（退票完成，座位释放） |

#### 2.6.2 订单/销售单状态（Sale）

| 值 | 标识 | 含义 |
| --- | --- | --- |
| `0` | `unpaid` | 待支付 |
| `1` | `paid` | 已支付 |
| `2` | `refunding` | 退票处理中 |
| `3` | `refunded` | 已退票（全额/部分） |
| `4` | `cancelled` | 已取消（超时未支付） |

#### 2.6.3 销售类型

| 值 | 标识 | 含义 |
| --- | --- | --- |
| `1` | `online` | 网络购票（观众端） |
| `2` | `offline` | 线下售票（售票员） |

#### 2.6.4 状态流转图

```text
票生命周期：
  available → locked → sold → checked
                  ↓        ↓
              (超时释放)   refunding → refunded → available

订单生命周期：
  unpaid → paid → refunding → refunded
     ↓
  cancelled
```

> **关键规则**：
> - `scheduleId` + `seatId` 唯一确定一张票（`ticketId`）。前端选座位，后端按此映射锁定对应票。
> - 锁定时长默认 **5 分钟**，超时自动释放（`locked` → `available`）。
> - 已验票（`checked`）不可退票。退票必须在演出开始前发起。
> - 退票后座位释放，该票可被重新购买。

#### 3.1.6 设置座位

```
PUT /admin/api/studios/:id/seats
```

**请求体**：

```json
{
  "seatRows": [
    "aaaa_aaa",
    "aaaa_aaa",
    "aaaa_aaa",

  ]

```

> 座位格式说明：`a` 表示可用座位，`_` 表示过道/不可用。每个字符对应一个座位位置。

**响应 data**：

```json
{
  "studioId": 1,
  "rowCount": 8,
  "colCount": 8,
  "seats": [
    { "id": 1, "row": 1, "col": 1, "status": 0 },
    { "id": 2, "row": 1, "col": 2, "status": 0 },
    { "id": 3, "row": 1, "col": 3, "status": 0 },
    { "id": 4, "row": 1, "col": 4, "status": 0 },
    { "id": 5, "row": 1, "col": 5, "status": -1 }
  ],
  "layout": [
    "aaaa_aaa",
    "aaaa_aaa",
    "aaaa_aaa",
    "aaaa_aaa",
    "aaaa_aaa",
    "aaaa_aaa",
    "aaaa_aaa",
    "aaaa_aaa"
  ]
}
```

> `status`：`0` 可用，`-1` 不可用（过道/损坏）。此接口会同步生成或禁用对应的 `Seat` 记录，前端可直接用返回数据刷新座位图。

### 3.3 演出计划管理（Schedule）

#### 3.3.1 查询演出计划列表

```
GET /admin/api/schedules?playId=&studioId=&startDate=&endDate=&page=1&pageSize=10
```

**响应 data.list 元素**：

```json
{
  "id": 1,
  "playId": 1,
  "playName": "雷雨",
  "studioId": 1,
  "studioName": "1号厅",
  "showTime": "2026-06-15 19:30:00",
  "ticketPrice": 180.00,
  "status": 1
}
```

#### 3.3.2 查询单个演出计划

```
GET /admin/api/schedules/:id
```

#### 3.3.3 新增演出计划

```
POST /admin/api/schedules
```

**请求体**：

```json
{
  "playId": 1,
  "studioId": 1,
  "showTime": "2026-06-15 19:30:00",
  "ticketPrice": 180.00
}
```

**业务校验**：

- 演出厅在该时段不可被其他计划占用（冲突检测）
- 演出时间必须晚于当前时间

**副作用**：

- 新增演出计划成功后，系统自动基于该演出厅的座位布局，为该场演出批量生成演出票（`Ticket`），初始状态为 `available`
- 票的默认价格取自演出计划中设定的 `ticketPrice`

#### 3.3.4 修改演出计划

```
PUT /admin/api/schedules/:id
```

#### 3.3.5 删除演出计划

```
DELETE /admin/api/schedules/:id
```

#### 3.3.6 查询某场演出的票/座位状态

```
GET /admin/api/schedules/:id/tickets?status=&page=1&pageSize=200
```

**响应 data.list 元素**：

```json
{
  "ticketId": 101,
  "seatId": 1,
  "seatRow": 1,
  "seatCol": 1,
  "price": 180.00,
  "status": 0,
  "lockTime": null
}
```

> 售票员通过此接口获取可售票列表，选中票后传给售票接口。`status=0` 筛选可售票。

#### 3.4.1 查询验票记录

```
GET /admin/api/checks?scheduleId=&ticketId=&page=1&pageSize=10
```

#### 3.4.2 验票

```
POST /admin/api/tickets/:ticketId/verify
```

**请求体**（可选）：

```json
{
  "operatorId": 1
}
```

**响应 data**：

```json
{
  "ticketId": 123456,
  "seatRow": 3,
  "seatCol": 5,
  "playName": "雷雨",
  "studioName": "1号厅",
  "showTime": "2026-06-15 19:30:00",
  "status": "checked",
  "message": "验票通过"
}
```

**业务校验**：

- 票是否存在
- 票状态必须为 `sold`（已售），不可重复验票
- 当前时间与演出时间偏差不超过 ±2 小时

### 3.8 票务管理（Ticket & Sale）

#### 3.8.1 售票（线下）

```
POST /admin/api/sales
```

**请求体**：

```json
{
  "scheduleId": 1,
  "ticketIds": [101, 102],
  "customerId": 1,
  "paymentAmount": 360.00
}
```

**响应 data**：

```json
{
  "saleId": 20260601001,
  "change": 0.00
}
```

#### 3.8.2 查询销售记录

```
GET /admin/api/sales?keyword=&type=&startDate=&endDate=&employeeId=&page=1&pageSize=10
```

**响应 data.list 元素**：

```json
{
  "id": 20260601001,
  "employeeName": "张三",
  "customerName": "李四",
  "saleTime": "2026-06-01 14:30:00",
  "paymentAmount": 360.00,
  "change": 0.00,
  "type": 1,
  "saleType": 1,
  "status": 1,
  "items": [
    {
      "id": 1,
      "ticketId": 101,
      "seatRow": 3,
      "seatCol": 5,
      "price": 180.00
    }
  ]
}
```

#### 3.8.3 退票

```
POST /admin/api/sales/:id/refund
```

**请求体**：

```json
{
  "ticketIds": [101]
}
```

**响应 data**：

```json
{
  "saleId": 20260601001,
  "refundedTickets": [101],
  "refundAmount": 180.00,
  "orderStatus": "refunded",
  "ticketStatus": "refunded"
}
```

**业务校验与规则**：

- 订单状态必须为 `paid`
- 票状态必须为 `sold`（已售），不可为 `checked`（已验票）或已在退票中
- 演出开始前可退；演出开始后原则上不可退
- 支持部分退票：从 `sale.items` 中移除对应记录，票状态变为 `refunded` → `available`
- 该规则同时适用于管理端线下退票和观众端线上退票

```json
  "showTime": "2026-06-15 19:30:00",
  "ticketPrice": 180.00,
  "seats": [
    {
      "id": 1,
      "row": 1,
      "col": 1,
      "status": 0
    }
  ],
  "seatLayout": [
    "aaaa_aaa",
    "aaaa_aaa",
    "aaaa_aaa",
    "aaaa_aaa",
    "aaaa_aaa",
    "aaaa_aaa",
    "aaaa_aaa",
    "aaaa_aaa"
  ]
}
```

> 座位 status 对应票状态枚举：`0` = available（可选），`1` = locked（锁定中），`2` = sold（已售）

### 4.3 选座与购票

#### 4.3.1 锁定座位

```
POST /customer/api/orders/lock
```

**请求体**：

```json
{
  "scheduleId": 1,
  "seatIds": [1, 2]
}
```

> 后端按规则 `scheduleId + seatId → ticketId` 映射并锁定对应的票。前端选的是座位，后端锁定的是票。

**响应 data**：

```json
{
  "lockToken": "uuid-lock-token",
  "tickets": [
    { "ticketId": 101, "seatId": 1, "row": 1, "col": 1, "price": 180.00 },
    { "ticketId": 102, "seatId": 2, "row": 1, "col": 2, "price": 180.00 }
  ],
  "totalPrice": 360.00,
  "expireAt": "2026-06-10T14:35:00Z"
}
```

**业务校验**：

- 座位对应的票状态必须为 `available`
- 锁定时长 **5 分钟**，超时自动释放（`locked` → `available`）
- 同一用户对同一场次最多锁定 6 张票

#### 4.3.2 下单

```
POST /customer/api/orders
```

**请求体**：

```json
{
  "lockToken": "uuid-lock-token"
}
```

> 后端根据 `lockToken` 获取已锁定的票列表，创建订单。不再重复传 `seatIds` 或 `ticketIds`。

**响应 data**：

```json
{
  "orderId": 20260610001,
  "totalPrice": 360.00,
  "status": "unpaid",
  "expireAt": "2026-06-10T15:00:00Z"
}
```

**业务校验**：

- `lockToken` 有效且未过期
- 对应的票状态仍为 `locked`（未被超时释放）

#### 4.3.3 支付

```
POST /customer/api/orders/:id/pay
```

**请求体**：

```json
{
  "paymentMethod": "balance",
  "paymentPassword": "123456"
}
```

> `paymentMethod` 可选值：`balance`（余额）、`wechat`（微信）、`alipay`（支付宝）

**响应 data**：

```json
{
  "orderId": 20260610001,
  "orderStatus": "paid",
  "tickets": [
    {
      "ticketId": 101,
      "seatRow": 1,
      "seatCol": 1,
      "playName": "雷雨",
      "studioName": "1号厅",
      "showTime": "2026-06-15 19:30:00",
      "price": 180.00,
      "ticketStatus": "sold"
    }
  ]
}
```

**业务校验**：

- 订单状态必须为 `unpaid`
- 支付密码校验（余额支付时）
- 支付超时时间 15 分钟，超时则订单取消（`unpaid` → `cancelled`），票释放（`locked` → `available`）

#### 4.3.4 查询订单状态

```
GET /customer/api/orders/:id
```

## 5. Mock 数据策略

### 5.1 方案

前端使用 **MSW（Mock Service Worker）** 或 **本地 JSON Server** 拦截 HTTP 请求并返回模拟数据。

推荐使用 MSW：无需启动额外服务，在浏览器端拦截请求，开发体验好。

### 5.2 Mock 目录结构

```
src/mocks/
  ├── handlers/           # 按模块拆分的 mock 处理器
  │   ├── studio.ts
  │   ├── play.ts
  │   ├── schedule.ts
  │   ├── sale.ts
  │   ├── employee.ts
  │   ├── customer.ts
  │   ├── finance.ts
  │   ├── auth.ts
  │   └── index.ts        # 汇总导出
  ├── data/               # Mock 静态数据
  │   ├── studios.json
  │   ├── plays.json
  │   ├── schedules.json
  │   ├── employees.json
  │   └── customers.json
  ├── browser.ts          # 浏览器端入口
  └── server.ts           # Node 端入口（可选）
```

### 5.3 Mock 数据要求

- 每条 mock 数据需覆盖正常响应（`10000`）和至少一种异常响应
- 列表接口需支持分页参数模拟
- 写操作（POST/PUT/DELETE）需模拟数据变更效果
- 登录接口需模拟 token 生成与验证
